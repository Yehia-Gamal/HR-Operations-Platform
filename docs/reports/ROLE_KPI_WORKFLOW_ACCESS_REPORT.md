# Role/KPI Workflow Access Report — v1.2.0

تم تنفيذ إعادة ضبط منطق الأدوار ومسار تقييم KPI في نسخة الويب فقط.

## ما تم تنفيذه

1. جعل تطبيق الموظفين متاحًا للتسجيل الذاتي؛ أي تسجيل جديد يتم إنشاؤه كموظف بصلاحيات Employee فقط.
2. إضافة صفحة KPI داخل تطبيق الموظفين باسم "تقييمي".
3. ضبط مسار KPI:
   - الموظف يرفع التقييم الذاتي.
   - المدير المباشر يعتمد/يعدّل تقييم فريقه فقط.
   - HR يضيف/يراجع جزء الحضور والانصراف والملفات الخاصة بالموارد البشرية.
   - السكرتير التنفيذي/التقني يراجع ويرفع.
   - المدير التنفيذي يعتمد النتيجة النهائية.
4. جعل المديرين يرون التيم الخاص بهم حسب `managerEmployeeId`.
5. فصل HR عن الصلاحيات التقنية؛ HR يرى الموارد البشرية والحضور والتقييمات والطلبات فقط.
6. جعل السكرتير التنفيذي/التقني كامل الصلاحيات على النظام.
7. ضبط لجنة حل المشاكل والخلافات:
   - السكرتير التنفيذي.
   - مدير تشغيل 1.
   - مدير تشغيل 2.
   - مدير الموارد البشرية.
   - التصعيد للمدير التنفيذي عبر السكرتير التنفيذي.
8. إضافة Patch جديد:
   `038_kpi_cycle_control_reports.sql`
9. إضافة فحص جديد:
   `tools/check-role-kpi-workflow.mjs`

## الفحوصات

```bash
bash tools/run-checks.sh
```

## ملاحظات النشر

عند الرفع الحقيقي على Supabase يجب تشغيل SQL حتى:

```text
038_kpi_cycle_control_reports.sql
```

ويجب تنفيذ Backend حقيقي لمسار `/auth/register-employee` في حال الاعتماد على API/Supabase بدل Local fallback.


## تسجيل الموظفين الحقيقي عبر Supabase

تمت إضافة Edge Function باسم `employee-register` حتى يكون تسجيل الموظف من تطبيق الموظفين حقيقيًا على Supabase، وليس fallback محلي. بعد التسجيل يتم إنشاء Auth User + Employee + Profile بدور `employee` ثم تسجيل الدخول تلقائيًا من الواجهة.
